# 人工签名合成 App · 施工细则（plan.md）

## 0. 需求定稿（已与用户确认）

- **形态**：macOS 桌面应用，Electron 界面
- **多用户**：支持多用户，每用户独立签名库
- **建库**：上传签名照片 → 标注签名对应的词 → **逐字画框**（拖鼠标画矩形框，松手确认，顺序对应字序，可撤销上一框）→ 单字图入库
- **图像风格**：全程纯白底 + 纯黑字，笔画二值化并适度加粗（入库、预览、插入表格统一）
- **合成策略**：整词优先——表格词与库里完整签名一致时直接贴整张签名图；无整词则单字拼接（统一字高、相邻字重叠 10%、白底正片叠底融合）
- **缺字处理**：缺笔迹的字保留系统字体混排 + 标红提醒补录（两者都要）
- **写回方式**：合成签名图直接替换单元格内文字（清掉文字，图片悬浮盖在单元格位置）；签名图宽高比锁定、等比放大至充满单元格（取整个合并区域尺寸，仅留约 4% 安全边）
- **预览页**：合成后先出预览页，每个签名渲染在按真实单元格（含合并区域）宽高比的模拟框内，所见即导出所得；每格可缩放大小、调对齐位置（左/中/右 × 上/中/下）、换一张字图（同字多笔迹切换）、重新生成；全部确认后统一导出
- **字体库可视列表**：单字/整词缩略图网格展示；每个条目可单独删除；同字多图可删单张、设默认用哪张；支持直接上传单字图片补库
- **表格编辑**：app 内做基础编辑（打开 xlsx、查看、填姓名、执行转换、保存）；复杂排版仍在 Excel/WPS 完成

## 1. 施工红线（不可逾越）

1. **不改需求定稿范围**：第一版不做全自动 AI 识别、不做连笔笔迹生成、不做复杂表格编辑器。任何"顺手加上"的功能都属于越界。
2. **不引入 Python 依赖**：图像处理与 xlsx 读写全部使用 Node 生态，避免打包时携带 Python 运行时。
3. **不破坏原始文件**：转换结果默认**另存为新文件**（`原名_已合成.xlsx`），绝不在未确认的情况下覆盖用户原文件。
4. **数据不出本机**：签名照片、签名库、表格文件全部存储在本机应用数据目录，不联网、不上传、不引入任何遥测。
5. **签名库必须可备份**：签名库以"普通文件目录 + JSON 索引"形式存储，用户可直接拷贝整个目录完成备份/迁移；禁止用不透明二进制数据库。
6. **多用户隔离**：切换用户后，界面与文件路径必须完全隔离，禁止跨用户读到他人签名库。
7. **白底黑字全程统一**：所有单字图/整词图入库前统一二值化为纯白底 + 纯黑字 PNG 并适度加粗笔画；合成与插入表格全程保持白底位图，禁止中途改回透明底或输出灰底图。
8. **不修改用户原始文件**：开发与验收使用临时目录中自动生成的全合成样例，任何用户选择的表格都不得被覆盖。

## 2. 技术方案

### 2.1 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 桌面壳 | Electron（最新稳定版） | 用户明确要求"像 app"的界面 |
| 前端 | 原生 HTML/CSS/JS + Electron 渲染进程，不引框架 | 界面规模小，保持简单 |
| 图像处理 | `sharp` | 裁剪、缩放、透明通道、像素级混合均在 Node 内完成 |
| 表格读写 | `exceljs` | 支持读写 xlsx、读写单元格、插入图片（twoCellAnchor 悬浮图） |
| 表格展示 | 渲染进程内自绘只读表格网格（HTML table）+ 可编辑单元格 | 满足"基础编辑"即可，不引重型表格库 |
| 数据存储 | 文件系统目录 + JSON 索引 | 红线 5 |

**已识别的技术风险与对策**：`exceljs` 读入再另存复杂 xlsx 时可能丢失部分它不支持的特性（如某些条件格式、图表）。对策：开发第一阶段就用样例文件做往返测试（读入→另存→对比），若丢失严重，备选方案为直接操作 xlsx 的 XML（zip 内改写 sheet/drawing 部分），该备选封装在同一模块接口后，不影响上层代码。

### 2.2 目录结构（项目）

```
sign-compose-app/
├── package.json
├── src/
│   ├── main/                  # Electron 主进程
│   │   ├── index.js           # 窗口创建、IPC 注册
│   │   ├── users.js           # 用户管理（增删切换）
│   │   ├── library.js         # 签名库读写（JSON 索引 + PNG 文件）
│   │   ├── image.js           # 图像处理（去白底、裁剪、单字切分建议、拼接融合）
│   │   └── xlsx.js            # 表格读写、文字替换、图片插入
│   ├── preload.js             # contextBridge 安全桥
│   └── renderer/              # 界面（原生 JS）
│       ├── index.html
│       ├── app.js
│       ├── views/
│       │   ├── users.js       # 用户选择/创建
│       │   ├── library.js     # 建库向导：上传→框选→核对→入库
│       │   └── workspace.js   # 工作区：打开表格→编辑→转换→保存
│       └── styles.css
└── userData/                  # 运行时生成（签名库，见 2.3）
```

### 2.3 签名库数据格式（每用户一份）

```
userData/
└── users/
    └── <用户id>/
        ├── profile.json          # { id, name, createdAt }
        ├── library.json          # 索引，见下
        ├── chars/                # 单字图：<汉字>_<序号>.png（透明背景）
        └── words/                # 整词原图：<词语>_<序号>.png（透明背景）
```

`library.json` 结构：

```json
{
  "chars": {
    "甲": [{ "id": "char_<UUID>", "file": "chars/char_<UUID>.png", "sourceWord": "甲乙丙" }]
  },
  "words": {
    "甲乙丙": [{ "id": "word_<UUID>", "file": "words/word_<UUID>.png" }]
  }
}
```

- 一个字允许存多张图，合成时默认取第 1 张（后续可扩展随机选取，第一版不做）。

### 2.4 核心流程设计

**A. 建库向导（renderer/views/library.js + main/image.js）**
1. 用户上传签名照片 + 输入该签名对应的词（文档示例统一使用虚构占位符“甲乙丙”）。
2. 主进程用 sharp 读图：灰度化 → 阈值二值化 → 白底转透明。
3. **自动切分建议**：对二值图做垂直投影分析，在笔画间隙处给出建议切割线（投影低谷）；连笔处允许切割线穿过浅连接。
4. 渲染进程在画布上显示图片 + 建议切割线，用户可拖动调整；每个框自动按顺序对应所标词语的第 N 个字，用户可修正对应关系。
5. 确认后：每个框裁出单字 PNG 存 `chars/`，整张签名（去白底）存 `words/`，更新 `library.json`。

**B. 工作区（renderer/views/workspace.js + main/xlsx.js）**
1. 打开 xlsx → exceljs 读入 → 渲染为可编辑网格（显示文本内容；不还原全部复杂排版，只保证数据正确）。
2. 用户在网格里填写/修改单元格文本（主要是姓名）。
3. 点击"合成签名"：用户指定（或 app 扫描全部）需要转换的单元格。
4. 对每个目标单元格的词：
   - 查 `library.json.words` → 命中则取整词图；
   - 未命中则逐字查 `chars`：全部命中 → 单字拼接（每字等比缩放到统一字高，横向排列，相邻字重叠 10% 并做 alpha 渐变过渡）；
   - 部分缺字 → 缺字用系统字体渲染成图插入混排，并在界面结果列表**标红提示缺字**，同时该单元格标红边框。
5. 转换：清空单元格文本 → 用 `worksheet.addImage()` 以 twoCellAnchor 方式把合成 PNG 悬浮锚定在单元格位置，尺寸自适应单元格（高度 ≈ 行高 80%）。
6. 另存为 `原名_已合成.xlsx`（红线 3）。

### 2.5 实施步骤（按序施工）

1. **脚手架**：Electron 工程初始化、主/渲染进程 IPC 骨架、窗口与导航（用户选择页 / 建库页 / 工作区页）。
2. **用户与签名库模块**：`users.js` + `library.js` + 数据目录结构，含单元级自测脚本。
3. **图像模块**：`image.js` 二值化（白底黑字+加粗）、区域裁剪、白边修剪、拼接融合；用样例签名图跑通命令行自测。
4. **建库向导界面**：逐字画框交互（拖矩形、撤销、字序对应），入库闭环；字体库可视列表（缩略图、删除、设默认、补单字）。
5. **xlsx 模块**：读入/网格化、写回、图片锚定插入（适配单元格）、另存；**此步必须先用样例文件做往返测试**（见 2.1 风险对策）。
6. **工作区界面**：表格展示与编辑、预览页（缩放/对齐/换字图/重新生成）、统一导出、缺字标红。
7. **联调与验收**：按第 3 节验收标准逐项过。

## 3. 验收标准

### 3.1 功能验收（必须全部通过）

| # | 验收项 | 通过标准 |
|---|---|---|
| 1 | 多用户 | 创建 ≥2 个用户，各自上传签名建库，切换用户后互不可见对方数据 |
| 2 | 建库闭环 | 上传一张签名照（3 字词），逐字画完 3 个框后入库：`chars/` 生成 3 张白底黑字单字图、`words/` 生成 1 张整词图、`library.json` 索引正确（含默认图标记） |
| 3 | 画框交互 | 拖鼠标画矩形框即得该字图像；画错可撤销上一框；框外白边自动裁掉 |
| 4 | 整词合成 | 表格单元格内容与库中整词一致时，输出文件中该单元格为整张签名悬浮图，无拼接痕迹 |
| 5 | 单字拼接 | 词不在库但所有单字在库时，输出为拼接签名图；相邻字衔接处无白缝、无突兀截断 |
| 6 | 缺字处理 | 词中有缺字时：缺字位置为系统字体渲染，app 界面结果列表标红提示具体缺哪个字，且该单元格在网格中标红 |
| 7 | 文字替换 | 转换后单元格原文本被清除，图片悬浮锚定在该单元格位置，行/列尺寸变化时图片随单元格移动（twoCellAnchor） |
| 8 | 原文件保护 | 转换后原 xlsx 文件字节不变（校验修改时间与哈希），输出为 `原名_已合成.xlsx` |
| 9 | 表格保真 | 用临时生成的全合成样例做转换，输出文件在 Excel/WPS 打开：表头、合并单元格、已有悬浮图片、说明文字均正常显示 |
| 10 | 白底黑字 | 入库图与输出表格中的签名图均为纯白底、纯黑字；笔画较原照片明显加粗 |
| 11 | 适配单元格 | 签名图宽高比锁定、等比放大至充满单元格（目标格在合并区时按整个合并区域计算），宽高均不超出、且充满方向达到区域尺寸 95% 以上 |
| 12 | 预览页 | 合成后先出预览页：每个签名显示在按真实单元格宽高比的模拟框内，调整缩放/对齐即时刷新；支持换字图、重新生成；确认后统一导出，导出前不产生任何文件 |
| 13 | 字体库列表 | 缩略图网格展示全部单字/整词；可逐个删除；同字多图可删单张、设默认（合成取默认图）；可直接上传单字图补库 |

### 3.2 工程验收

- `npm start` 一条命令可启动应用，无控制台报错。
- 签名库目录可直接拷贝到另一台机器对应位置，应用可识别（红线 5 验证）。
- 全程无网络请求（断网状态下功能完整可用）。

### 3.3 明确不做（本期验收不要求）

- 全自动手写汉字识别
- 真连笔笔迹生成 / 字间笔画拟合
- 同字多笔迹随机选取
- 表格复杂排版编辑（合并、样式、打印设置等）
- Windows/移动端打包
